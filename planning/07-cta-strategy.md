# CTA Strategy

CTA는 포지션 제안과 면접 제안을 가장 쉽게 만들기 위한 장치다. GitHub나 Blog 방문보다 PDF 다운로드와 이메일 연락을 우선한다.

## Primary CTA

### PDF 이력서 다운로드

Label candidates:

- 이력서 다운로드
- PDF 이력서 보기

Target:

- `이혜주_FrontendEngineer.pdf`

Placement:

- Hero
- Resume section
- Mobile sticky 또는 compact CTA는 디자인 단계에서 검토

주의:

- 사이트의 public identity는 Peppearl로 유지한다.
- PDF 파일명에는 실명이 포함되어 있지만 사용자가 공개 가능한 자료로 제공했다.

## Secondary Links

| Link | 역할 | 노출 강도 |
| --- | --- | --- |
| LinkedIn | professional profile 보조 확인 | 중간 |
| Blog | 글과 학습 흔적 확인, Case 01 Related Writing 보조 | 낮음-중간 |
| GitHub | 코드/계정 확인 보조 | 낮음 |

Wanted는 사용하지 않는다.

## GitHub가 메인처럼 보이지 않게 하는 방법

- Hero primary CTA에 GitHub를 넣지 않는다.
- GitHub stats, repo grid, contribution graph를 넣지 않는다.
- GitHub는 LinkedIn, Blog와 같은 secondary link 그룹에 둔다.
- Selected Work의 근거는 GitHub repo가 아니라 case narrative, decision, diagram으로 만든다.
- Blog도 Hero primary CTA로 올리지 않는다. 관련 글은 Selected Work 하단이나 footer에서만 연결한다.

## CTA 우선순위

1. 이력서 다운로드
2. 이메일로 연락하기
3. 대표 프로젝트 보기
4. LinkedIn
5. Blog
6. GitHub

## 모바일 CTA 원칙

- 첫 화면에서 최소 하나의 primary CTA가 보여야 한다.
- CTA 버튼은 줄바꿈되어도 읽기 쉬워야 한다.
- Contact section까지 내려가지 않아도 이메일 행동이 가능해야 한다.
- secondary links는 작은 화면에서 icon/text list로 축약할 수 있다.
