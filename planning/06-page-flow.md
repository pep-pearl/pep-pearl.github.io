# Page Flow

사이트는 채용 담당자가 빠르게 훑을 수 있어야 하고, 팀 리드가 더 깊게 읽을 때는 구조와 기술 판단이 보여야 한다.

## 기본 읽기 흐름

```txt
Hero
-> About
-> Core Strengths
-> Experience
-> Selected Work
-> Tech Stack
-> Resume
-> Contact
```

## 1분 스캔 흐름

대상: Recruiter

1. Hero에서 역할과 포지셔닝 확인
2. Core Strengths에서 강점 3개 확인
3. Experience에서 3년차 프론트엔드 개발자임을 확인
4. Resume 또는 Contact CTA로 이동

이 흐름에서는 긴 케이스 설명보다 요약 카드와 CTA가 중요하다.

## 3분 검토 흐름

대상: Frontend Lead, CTO, Founder

1. Hero에서 포지셔닝 확인
2. About에서 어떤 관점으로 일하는지 확인
3. Core Strengths에서 Design System, Architecture, AI Workflow 확인
4. Selected Work에서 실제 판단과 구현 방향 확인
5. Tech Stack에서 기술 사용 맥락 확인
6. Resume PDF 또는 Email로 이동

이 흐름에서는 GitHub보다 업무 경험과 문제 해결 과정이 먼저 보여야 한다.

## 깊게 읽는 흐름

대상: Frontend Lead, CTO

1. Selected Work의 각 case를 읽는다.
2. case별 decision과 implementation을 확인한다.
3. diagram, pseudo code, state matrix를 통해 기술 판단을 확인한다.
4. Tech Stack과 Experience를 교차 확인한다.
5. LinkedIn 또는 PDF로 이력 세부를 확인한다.

## CTA 연결 흐름

Hero:

- PDF 이력서 다운로드
- 대표 프로젝트 보기
- 이메일로 연락하기

Selected Work 이후:

- "더 자세한 경력은 PDF에서 확인하기"
- "프로젝트 관련 제안 보내기"
- Related Writing은 Case 01 하단에만 제한적으로 연결하고, 외부 블로그가 메인 흐름을 끊지 않게 한다.

Resume 이후:

- PDF 다운로드
- 이메일로 연락하기

Contact:

- Email primary
- LinkedIn, Blog, GitHub secondary

## GitHub 노출 위치

GitHub는 다음 위치에서만 보조 링크로 사용한다.

- Hero secondary links
- Contact secondary links
- 필요 시 footer

GitHub stats, contribution graph, repository list는 v1에서 사용하지 않는다.

## 이탈 방지 원칙

- 첫 화면에서 외부 링크보다 내부 anchor와 contact CTA를 우선한다.
- Selected Work 전에 GitHub로 빠져나가게 하지 않는다.
- PDF 다운로드는 hero와 resume section 두 곳에서 제공한다.
- Email CTA는 hero와 contact section 두 곳에서 제공한다.
- Blog는 footer 또는 selected work의 보조 링크로만 사용한다.
