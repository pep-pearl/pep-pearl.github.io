# Site Strategy

## 사이트 정의

Peppearl의 개인 커리어 사이트는 GitHub 중심 포트폴리오가 아니라, 포지션 제안과 면접 제안을 유도하는 커리어 랜딩페이지다. 이력서, 경력기술서, 포트폴리오를 하나의 흐름으로 묶어 방문자가 PDF를 열기 전에도 Peppearl의 강점과 경력 흐름을 이해할 수 있게 한다.

## 핵심 목적

- 채용 담당자와 기술 리더가 3분 안에 강점을 파악하게 한다.
- PDF 이력서 없이도 경력, 주요 프로젝트, 기술 방향성을 이해하게 한다.
- 면접관에게 링크 하나만 보내도 되는 커리어 자료로 만든다.
- 실무 경험, 구조적 사고, 디자인 시스템, 프론트엔드 아키텍처, AI Agent Workflow를 중심으로 설득한다.
- GitHub는 보조 근거로만 두고, 업무 경험과 문제 해결 과정을 먼저 보여준다.

## 핵심 포지셔닝

> 지속 가능한 디자인 시스템과 확장성 있는 아키텍처로 제품의 완성도를 높이는 프론트엔드 개발자

Hero에서는 아래 문장을 우선 사용한다.

> 제품의 완성도를 높이는 프론트엔드 개발자
>
> 디자인 시스템, 확장 가능한 아키텍처, AI Agent 활용 워크플로우를 바탕으로 복잡한 서비스를 안정적이고 세련되게 구현합니다.

## 주요 대상

1. Recruiter: 빠르게 역할, 경력, 연락 방법을 확인해야 한다.
2. Frontend Lead: 기술 판단, 구조화 역량, 유지보수 관점을 확인해야 한다.
3. CTO / Engineering Leader: 복잡한 업무 시스템을 안정적으로 다룰 수 있는지 확인해야 한다.
4. Startup Founder / Project Client: 제품 완성도를 높일 수 있는 협업 가능성을 확인해야 한다.

## 핵심 메시지 축

### Design System

UI를 단발성 화면이 아니라 반복 가능한 기준으로 정리하는 역량을 보여준다. tokens, components, states, layout rules, accessibility를 함께 다루는 방향을 강조한다.

### Frontend Architecture

React, TypeScript, Vite, Feature-Sliced Design, pnpm workspace / monorepo 경험을 바탕으로 화면, 상태, 도메인 로직, 공용 UI의 경계를 정리하는 역량을 보여준다.

### AI Agent Workflow

AI Agent를 단순 코드 생성기가 아니라 기획, 구현, 리뷰를 구조화하는 작업 파트너로 사용하는 방식을 보여준다. AGENTS.md, phase rules, prompt templates, review gates를 근거로 삼는다.

## 비목표

- GitHub 활동량을 전면에 내세우지 않는다.
- 토이 프로젝트 모음으로 만들지 않는다.
- 업무 화면 이미지나 Figma 캡처를 사용하지 않는다.
- 구체적인 성과 수치를 공개하지 않는다.
- 과장된 자기 PR, "최고", "완벽", "압도적" 같은 표현을 쓰지 않는다.
- 공공/SI 프로젝트 목록처럼 보이게 하지 않는다.
- 디자인만 하는 사람처럼 보이지 않게, 구현과 구조 판단을 함께 보여준다.

## 성공 기준

- 방문자가 3분 안에 Peppearl의 강점 세 가지를 이해한다.
- 이력서 PDF 없이도 경력 흐름과 대표 프로젝트가 파악된다.
- 대표 케이스 3개가 스크린샷이나 수치 없이도 설득력 있게 읽힌다.
- Resume download와 Email contact가 자연스럽게 이어진다.
- GitHub는 보조 링크로 보이고, 메인 CTA처럼 보이지 않는다.
- 모바일에서도 핵심 포지셔닝, CTA, 대표 강점이 먼저 보인다.

## 기획 판단

v1은 원페이지 구조가 적합하다. 콘텐츠 깊이가 아직 상세 페이지를 요구할 만큼 분리되어 있지 않고, 채용 담당자에게는 한 번에 훑는 흐름이 더 중요하다. 다만 Selected Work는 향후 상세 페이지로 확장할 수 있도록 케이스 구조를 보존한다.
