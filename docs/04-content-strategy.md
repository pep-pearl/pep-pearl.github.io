# 콘텐츠 전략

사이트 콘텐츠의 목적은 Peppearl의 경력과 강점을 빠르게 이해시키는 것이다. 이력서 PDF를 열지 않아도 역할, 경력 흐름, 대표 프로젝트, 연락 방법이 파악되어야 한다.

## 콘텐츠 방향

- 한국어를 기본으로 한다.
- 기술 키워드는 English term을 자연스럽게 병행한다.
- 과장하지 않는다.
- 겸손함은 직접 말하지 않고, 학습 태도와 작업 방식으로 드러낸다.
- GitHub 개인 작업보다 실무 프로젝트와 구조적 사고를 강조한다.
- 성과 수치가 부족한 부분은 부풀리지 않고 의사결정, 구조화, 문제 해결 과정으로 설득한다.

## 핵심 내러티브

```txt
Frontend Engineer
-> 제품의 완성도
-> Design System
-> Frontend Architecture
-> 복잡한 업무/GIS 인터페이스 경험
-> AI Agent Workflow
-> Resume / Contact
```

## 섹션별 역할

- Hero: 포지셔닝과 주요 CTA를 5초 안에 전달한다.
- About: 어떤 개발자인지 짧고 구체적으로 설명한다.
- Core Strengths: Design System, Frontend Architecture, AI Agent Workflow를 세 축으로 보여준다.
- Experience: PDF 없이도 경력 흐름을 이해하게 한다.
- Selected Work: 세 대표 케이스를 깊이 있게 보여주되, 기밀 이미지는 쓰지 않는다.
- Tech Stack: 기술을 나열하기보다 사용 맥락별로 묶는다.
- Resume: PDF 다운로드와 공개용 요약을 제공한다.
- Contact: 이메일과 외부 링크로 제안 행동을 쉽게 만든다.

## 선호 표현

- 구조화하다
- 기준을 세우다
- 유지보수하기 쉽게 만들다
- 사용자 흐름을 단순화하다
- 검증하고 다듬다
- 확장 가능한 구조
- 제품의 완성도를 높이다
- 화면, 상태, 도메인 로직의 경계를 정리하다
- 반복 가능한 작업 흐름을 만들다

## 피할 표현

- 최고
- 완벽
- 압도적
- 모든 문제를 해결
- 천재적
- 겸손한 사람
- AI로 다 해냄
- 무조건 빠르게
- 열정적인 개발자
- 사용자 경험을 최우선으로 하는 개발자라는 식의 근거 없는 일반론

## 권장 Hero 문장

```md
# 제품의 완성도를 높이는 프론트엔드 개발자

디자인 시스템, 확장 가능한 아키텍처, AI Agent 활용 워크플로우를 바탕으로
복잡한 서비스를 안정적이고 세련되게 구현합니다.
```

## 공개용 소개 톤

권장:

```md
React와 TypeScript를 중심으로 제품 UI의 기준과 구조를 함께 다루는 프론트엔드 개발자입니다.
디자인 시스템과 프론트엔드 아키텍처를 연결해 복잡한 서비스를 일관되고 유지보수하기 쉬운 형태로 만드는 데 관심이 있습니다.
```

피해야 할 방향:

- 긴 자기소개서 문체
- 감정적인 성장 서사
- 성과 수치 없는 과장
- GitHub 활동량 중심의 자기증명
- 업무 프로젝트를 공공/SI 목록처럼 나열하는 방식

## 대표 케이스 작성 방식

각 케이스는 아래 구조로 작성한다.

```txt
Context -> Problem -> My Role -> Key Decisions -> Implementation -> What I Learned
```

`Outcome`은 검증된 수치가 있을 때만 사용한다. 수치가 없다면 결정, 구현 방식, 제약, 학습을 중심으로 설득한다.

## 공공/SI 또는 업무 시스템 경험 표현 방식

도메인 자체보다 복잡한 제품 문제를 강조한다.

덜 좋은 표현:

```txt
공공 재난관리 시스템 구축
```

더 좋은 표현:

```txt
재난/GIS 도메인의 복잡한 업무 데이터를 사용자가 탐색하고 판단할 수 있는 인터페이스로 구조화한 경험
```

강조할 관점:

- 지도 기반 인터랙션
- 목록, 상세, 필터, 지도 상태의 연결
- 도메인 로직과 UI 경계
- 디자인 시스템과 컴포넌트 상태
- 유지보수 가능한 폴더 구조
- 접근성과 일관성

## AI Agent Workflow 표현 방식

AI를 결과를 대신 만드는 도구로 과장하지 않는다. 기준, 문서, 검토 루프를 가진 작업 파트너로 설명한다.

권장:

```md
AI Agent를 단순 코드 생성기가 아니라, 기획과 구현, 리뷰 과정을 구조화하는 작업 파트너로 사용합니다.
단계별 rule과 prompt, review gate를 통해 사람이 검토 가능한 기준을 먼저 세우는 데 집중합니다.
```

피할 방향:

- AI가 모든 개발을 대신한다는 인상
- 생산성 수치를 근거 없이 주장
- 사람의 판단을 대체한다는 표현
