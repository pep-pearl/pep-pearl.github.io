# Content Rules — Peppearl Career Site

## Purpose

Use this file when writing or editing all site content.

## Voice

Write in Korean first. Use English terms only when they are more natural or industry-standard.

Tone should be:
- Clear
- Calm
- Specific
- Professional
- Not exaggerated
- Not overly humble to the point of weakness
- Not loud or salesy

## Identity

Use: **Peppearl**

Do not use the user's real name as the main site identity unless the user explicitly changes this.

## Main Message

The site should consistently support this positioning:

> 지속 가능한 디자인 시스템과 확장성 있는 아키텍처로 제품의 완성도를 높이는 프론트엔드 개발자

Hero copy may use:

```md
제품의 완성도를 높이는 프론트엔드 개발자

디자인 시스템, 확장 가능한 아키텍처, AI Agent 활용 워크플로우를 바탕으로
복잡한 서비스를 안정적이고 세련되게 구현합니다.
```

## Content Priorities

Prioritize:
1. What Peppearl does well
2. How Peppearl thinks and decides
3. What kind of work Peppearl can handle
4. Evidence through structure, snippets, diagrams, and case decisions

Do not prioritize:
- GitHub activity
- Toy projects
- Unsupported achievements
- Vague passion statements
- Generic frontend slogans

## CTA Rules

Primary CTAs:
- 이력서 다운로드
- 대표 프로젝트 보기
- 메일 보내기

Secondary links:
- GitHub
- LinkedIn
- Wanted
- Blog

GitHub should be visible but not dominant.

## Privacy Rules

Do not include:
- Phone number
- Face photo
- Work screenshots
- Figma screenshots
- Confidential company/project details
- Unverified result numbers

Allowed:
- Email
- Alias
- Public links
- Resume PDF
- Code snippets
- Descriptive diagrams

## Case Study Format

Use this structure:

```txt
Context → Problem → My Role → Decisions → Implementation → What I Learned
```

### Context

Describe the work setting without exposing sensitive information.

### Problem

Describe the actual frontend/product problem. Avoid vague wording.

### My Role

Be direct about contribution. Do not exaggerate ownership.

### Decisions

Show judgment. This is more important than fake metrics.

### Implementation

Mention concrete tools, architecture choices, and implementation details.

### What I Learned

Use this to show humility and growth indirectly.

## Humility Rule

Do not say “I am humble.” Show it through:
- Learning notes
- Trade-off explanations
- Team-oriented wording
- “I focused on...” instead of “I led everything...”
- Mentioning constraints honestly

Good:
> 새로운 도메인을 빠르게 이해하고, 팀이 오래 유지할 수 있는 구조와 기준을 고민했습니다.

Bad:
> 저는 누구보다 겸손하고 열심히 일합니다.

## Claims Rule

Do not claim:
- Improved performance by X%
- Reduced time by Y%
- Increased quality by Z
- Led a project end-to-end

Unless the user provides verified evidence.

Instead use:
- “기여했습니다”
- “정리했습니다”
- “구조화했습니다”
- “기반을 마련했습니다”
- “일관되게 확장할 수 있도록 설계했습니다”

## Avoided Phrases

Avoid cliché portfolio phrases:
- “끊임없이 성장하는 개발자”
- “사용자 경험을 최우선으로 생각하는 개발자”
- “열정적인 개발자”
- “완벽을 추구하는 개발자”
- “문제를 해결하는 개발자” without concrete evidence

## Preferred Phrases

Use more grounded phrases:
- “복잡한 화면을 일관된 구조로 정리합니다.”
- “디자인 시스템과 프론트엔드 구현 사이의 간극을 줄입니다.”
- “팀이 유지보수할 수 있는 기준을 문서와 코드로 남깁니다.”
- “AI Agent를 코드 생성기가 아니라 검토 가능한 작업 파트너로 사용합니다.”
