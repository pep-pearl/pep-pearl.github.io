# 포트폴리오 정리본

## 1. FSD 도입을 통한 코드 구조 개선

### ❓ 왜 FSD(Feature-Sliced Design)를 사용했나요?

실무에서 Monorepo를 도입하는 과정에서 기존 구조의 한계가 드러났습니다.
대표적으로 **순환 참조 오류**와 **스파게티 코드** 문제가 있었고, 프로젝트 규모가 커질수록 코드 간 의존성이 복잡하게 얽히는 문제가 발생했습니다.

이를 해결하기 위해 **FSD(Feature-Sliced Design)** 구조를 도입했습니다.

초기에는 시행착오도 있었습니다.

- FSD 전용 ESLint 플러그인의 존재를 알지 못함
- 같은 레이어 간 참조를 지양해야 한다는 점을 명확히 인지하지 못함
- 팀원들의 러닝 커브 존재

하지만 도입 이후에는 복잡하게 얽혀 있던 코드 구조가 정리되었고, **스파게티 코드 문제와 복잡도 감소**에 효과가 있었습니다.

특히 동료들이 각 슬라이스의 이름과 Public API 역할을 하는 `index` 파일만 확인하면 되었기 때문에, 코드 탐색 비용이 줄어들었습니다.
그 결과 **개발 생산성**과 **만족도**가 올라갔고, 전반적인 **DX 개선**과 **캡슐화** 측면에서도 긍정적인 효과가 있었습니다.

---

## 2. INP 지표 개선 전략

### ❓ INP 지표 개선의 전략이 있는가?

INP 개선의 핵심은 **유저가 액션을 취했을 때 화면이 즉각적으로 반응하도록 메인 스레드 블로킹을 방어하는 것**이라고 생각합니다.

데이터 구조 변경이 발생했을 때도, 변경 범위가 불필요하게 상위 레이어까지 전파되지 않도록 관리하는 것이 중요했습니다.

### 데이터 변경 대응 전략

데이터의 **핵심 스펙은 유지**하면서, 페이징 등 외부 wrapper 구조가 변경되는 경우에는 변경 범위를 `shared` 또는 해당 쿼리가 정의된 파일 내부로 국한했습니다.
주로 `entities`, `shared` 레이어 내부에서 대응할 수 있도록 설계했습니다.

특히 `useInfiniteQuery`의 `select` 옵션을 **백엔드와 프론트엔드 UI 사이의 방어막**, 즉 **Anti-Corruption Layer** 역할로 활용했습니다.

예를 들어 `shared`의 API Client 응답 구조가 변경되더라도, `select` 함수 내부의 평탄화 로직만 변경하면 그 외 영역으로 변경이 전파되지 않도록 했습니다.

### 변경 범위에 따른 대응

도메인 데이터 자체의 스펙이 변경되는 경우에는 `entities` 레이어의 타입 수정이 불가피합니다.

반면 UI 변동이 없는 경우에는 `select` 옵션 내부에 Mapper 유틸 함수를 작성해 기존 인터페이스 형태로 변환하여 반환했습니다.
이렇게 하면 상위 레이어 수정이 필요하지 않습니다.

UI 변동이 있는 경우에는 `entities`, `features`, `widgets`까지 연쇄적으로 UI 리팩토링이 전파될 수 있다고 판단했습니다.

## 3. AI 협업 원칙과 활용 방식

### ❓ 이번 과제를 진행하면서 AI를 구체적으로 어떻게 활용했는가?

### 프롬프트 엔지니어링이나 AI 페어 프로그래밍 시 발생할 수 있는 환각(Hallucination)이나 구조적 훼손을 방지하는 본인만의 원칙이 있는가?

저는 AI를 단순한 코드 생성 도구가 아니라, **검증 가능한 파트너**로 활용하려고 했습니다.
특히 환각 방지, 구조적 훼손 방지, 효율성 극대화라는 세 가지 기준을 중심으로 AI를 사용했습니다.

### 1) 환각 방지: 교차 검증 및 상호 반론

**Multi-Role Cross-checking**

Antigravity(AI IDE)의 여러 에이전트와 Gemini CLI 등에 서로 다른 역할, 즉 **Persona**를 부여했습니다.

각 에이전트가 생성한 산출물에 대해 다른 에이전트가 공격적으로 반론하도록 구성했고, 이를 통해 **Red Teaming** 방식으로 환각을 걸러내도록 유도했습니다.

즉, 하나의 답변을 그대로 신뢰하기보다 서로 다른 관점의 AI가 상호 검증하도록 만들었습니다.

---

### 2) 구조적 훼손 방지: 선 설계 후 구현

**Planning-First Workflow**

구조적 훼손을 방지하기 위해 코딩부터 시작하지 않고, 먼저 **요구사항 정의서**와 **TDD 명세**를 확정했습니다.

고급 모델을 활용해 요구사항 분석, 인프라 계획, 테스트 코드(TDD) 명세를 작성했고, 이를 계획 문서 형태로 산출했습니다.

이후 산출물 분석은 사람이 직접 검토했습니다.
즉, **Human-in-the-loop**, 사람 중심 제어 방식을 유지했습니다.

AI가 작성한 계획을 그대로 구현하지 않고, 직접 문서를 검토하고 코멘트를 달아 계획을 수정하는 과정을 반복했습니다.

---

### 3) 효율성 극대화: 모델 티어링

**Model Tiering**

모든 작업에 고성능 모델을 사용하는 대신, 작업의 성격에 따라 모델을 나누어 사용했습니다.

- 고급 모델: 전략 수립과 설계
- 하위 모델: 단순 구현 반복 작업

전체 흐름은 다음과 같이 가져갔습니다.

> 중간 성능 모델로 연구
> → 고성능 모델로 계획 수립
> → 인간 검토 및 피드백
> → 고성능 모델로 계획 수정
> → 고성능 모델로 최종 계획 작성
> → 저성능 모델로 코딩하거나 직접 코딩

최종 계획은 “따라하기만 하면 되는 수준”으로 아주 구체적으로 만들고, 이후 구현 단계에서는 더 낮은 모델을 활용하거나 직접 코딩했습니다.

### AI 협업 원칙 요약

> 저는 AI를 단순 도구가 아닌 **검증 가능한 파트너**로 활용합니다.
> 상호 반론을 통해 환각을 잡고, 선 설계 후 구현 원칙으로 구조적 안정성을 유지하며, 모델 티어링으로 효율성까지 챙기는 것이 저만의 AI 협업 원칙입니다.

# 4. Micro Frontends(MFE) Deep Dive & 실무 회고

## 개요

FSD(Feature-Sliced Design)의 한계를 넘어서고, 대규모 팀의 독립적 배포와 개발 효율성을 극대화하기 위한 MFE 아키텍처와 전략을 정리했습니다.

---

## 4-1. MFE vs Monolith: 왜 넘어가는가?

| 구분      | Monolith with FSD                       | Micro Frontends                                       |
| --------- | --------------------------------------- | ----------------------------------------------------- |
| 격리 수준 | 논리적 격리 Layer/Slice                 | 물리적 격리 Repo, Build, Deploy 단위 분리             |
| 배포 단위 | 전체 프로젝트 단일 배포                 | 서비스/기능 단위 독립 배포                            |
| 순환 참조 | 프로젝트 규모가 커지면 발생 가능        | 물리적 분리로 인해 발생 원천 차단                     |
| 장점      | 초기 설정 단순, 공유 자원 접근 용이     | 팀 간 간섭 최소화, 빠른 배포 사이클                   |
| 단점      | 수정 시 전체 빌드/배포로 배포 병목 발생 | 초기 인프라 복잡성, 버전 관리 및 공유 설정이 까다로움 |

### 💡 나의 경험에서 얻은 인사이트

FSD는 레이어와 슬라이스로 논리적 분리를 제공하지만, 프로젝트가 거대해지면 슬라이스 간 데이터 얽힘이 발생할 수 있습니다.
또한 전체 빌드 시간이 기하급수적으로 늘어나면서 배포 병목이 생길 수 있다고 보았습니다.

pnpm 패키지화 방식과도 차이가 있습니다.

사내 서버를 통해 패키지화하는 Build-time 방식은 결국 어느 한 곳이 업데이트되면 다른 곳도 재빌드/재배포해야 하는 굴레를 벗어나기 어렵습니다.

반면 MFE의 Runtime 방식은 수정된 JS만 CDN에서 갈아끼우는 방식이기 때문에, 진정한 의미의 **독립 배포**가 가능하다고 판단했습니다.

---

## 4-2. 통합 방식의 핵심: Build-time vs Runtime

MFE를 구현할 때는 “언제 합치느냐”가 아키텍처의 성격을 결정합니다.

### 🏗️ Build-time Integration

Build-time 통합은 npm 또는 pnpm 패키지로 설치하여 빌드 시점에 포함하는 방식입니다.

타입 안정성이 높고 최적화가 쉽다는 장점이 있지만, 결합도는 여전히 높습니다.
예를 들어 A 패키지를 수정하면 B 서비스도 재빌드가 필요할 수 있습니다.

---

### ⚡ Runtime Integration

Runtime 통합은 브라우저에서 실행 중에 필요한 JS 번들을 불러와 결합하는 방식입니다.
진정한 의미의 MFE에 더 가깝다고 볼 수 있습니다.

주요 기술로는 `iframe`과 `Module Federation`이 있습니다.

`iframe`은 가장 완벽한 격리를 제공합니다.
외부 환경의 CSS나 전역 변수로부터 자유롭지만, 성능, 메모리, SEO, UX 측면에서 불리할 수 있습니다.
예를 들어 모달 처리와 같은 사용자 경험 구현이 까다로울 수 있습니다.

`Module Federation`은 Webpack 5의 기술로, 앱 간 의존성을 공유하면서 필요한 모듈만 원격으로 로드할 수 있습니다.

Runtime 통합의 가장 큰 장점은 배포 효율성입니다.
수정된 기능의 JS만 배포하면, 사용자는 새로고침만으로 업데이트된 내용을 확인할 수 있습니다.

---

## 4-3. MFE 핵심 운영 및 기술 전략

### 🛠️ 의존성 및 버전 관리

MFE에서는 중복 제거와 독립성 사이의 균형이 중요합니다.

React와 같은 공유 자원은 최대한 싱글톤으로 관리하여 번들 크기를 줄여야 UX 측면에서 이득이 있습니다.

또한 `requiredVersion`이나 싱글톤 경고를 통해 팀들이 버전을 맞추도록 유도할 수 있습니다.

다만 가혹한 규칙은 DX를 해치지만, 무분별한 예외는 시스템 전체를 망칠 수 있습니다.
따라서 에러 레벨에서 강제성을 부여하는 **하네스 엔지니어링(Harness Engineering)**적 접근이 필요하다고 보았습니다.

---

### 🔄 상태 공유

상태 공유에서는 데이터의 성격에 따라 전략을 달리해야 합니다.

Auth, User Profile처럼 무결성이 중요한 데이터는 물리적으로 하나의 객체로 관리하는 것이 논리적입니다.

반면 서비스 간 결합도를 낮추기 위해서는 전체 객체를 넘기기보다 `ID` 값 위주로 전달하는 것이 좋다고 판단했습니다.
이렇게 하면 업데이트 과정에서 발생할 수 있는 오류를 줄일 수 있습니다.

통신 방식은 단순한 데이터의 경우 URL이나 Window 이벤트를 사용할 수 있고, 복잡한 상태는 공유 상태 라이브러리를 활용하는 방식이 적절하다고 보았습니다.

---

## 4-4. 실전 마이그레이션 가이드라인

### 📅 점진적 마이그레이션

모놀리스에서 MFE로 전환할 때 가장 중요하게 고려해야 할 기술적 가이드라인은 **서비스 연속성을 보장하는 점진적 마이그레이션**이라고 생각합니다.

처음부터 전체를 바꾸는 것은 오버 엔지니어링이 될 수 있습니다.
따라서 가장 복잡하고 배포 병목이 심한 부분부터 부분 적용하는 방식이 리스크를 줄일 수 있습니다.

사용자는 내부 구조를 몰라도 됩니다.
서비스 운영에 지장이 없는 선에서 점진적으로 전환하는 것이 가장 상품성이 높은 접근이라고 생각합니다.

---

### 🚫 도입을 지양해야 할 때

MFE는 항상 정답이 아니라고 생각합니다.

먼저 플랫폼 엔지니어링, 즉 인프라를 지원할 여력이 없다면 배보다 배꼽이 더 커질 수 있습니다.

또한 규모가 작은 프로젝트에서는 굳이 분리할 필요가 없습니다.

마지막으로 기능들이 애초에 너무 끈끈하게 연결되어 있다면, 독립 배포의 이점을 얻기 어렵기 때문에 MFE 도입 효과가 제한적일 수 있습니다.

---

### 💬 면접용 한 줄 요약

> MFE는 단순히 기술적인 분리가 아니라, **팀 간의 독립적인 배포 사이클을 확보하여 비즈니스 민첩성을 높이는 전략**입니다.
> 저는 모노리스에서 겪었던 배포 병목과 순환 참조 문제를 해결하기 위해 Runtime 통합, 특히 Module Federation을 고려했으며, 이 과정에서 생기는 버전 충돌 문제는 기술적 가드레일을 통해 DX와 안정성을 동시에 잡아야 한다고 생각합니다.

# 5. Single Flight Pattern & Token Race Condition 해결 전략

## 5-1. Single Flight Pattern이란?

**Single Flight Pattern**은 동일한 요청이 동시에 여러 개 발생했을 때, 이를 하나로 묶어 처리하고 그 결과, 즉 응답을 공유하는 디자인 패턴입니다.

핵심 가치는 다음과 같습니다.

- 중복 리소스 낭비 방지
- 서버 부하 감소
- 데이터 일관성 유지

프론트엔드에서는 여러 탭이 동시에 Refresh Token을 사용하여 Access Token을 갱신하려고 할 때, 단 하나의 탭만 요청을 수행하게 하는 전략으로 사용할 수 있습니다.

---

## 5-2. SharedWorker vs BroadcastChannel

| 구분      | SharedWorker                                         | BroadcastChannel + Storage                      |
| --------- | ---------------------------------------------------- | ----------------------------------------------- |
| 구조      | 중앙 집중형                                          | 분산 협력형                                     |
| 특징      | 물리적/논리적으로 단일한 Worker가 관리               | 탭 간 이벤트 메시징을 통한 동기화               |
| 장점      | 상태가 한 곳에 있어 싱크 구현이 매우 단순함          | 리소스 점유가 적고 구현이 유연함                |
| 단점      | 별도의 Worker 리소스 필요, 브라우저 호환성 고려 필요 | 동기화 로직과 Race Condition을 직접 설계해야 함 |
| 선택 기준 | 데이터 무결성이 최우선일 때                          | 리소스 효율성과 브라우저 호환성이 중요할 때     |

---

## 5-3. 나의 해결 전략: BroadcastChannel + LocalStorage

브라우저 탭 간 통신을 위해 `BroadcastChannel API`를 사용하고, 토큰 상태 및 대장(Leader) 확인을 위해 `LocalStorage`를 공유 저장소로 활용했습니다.

### 구현 핵심 로직

1. 요청의 종류를 Key로 지정합니다.
   예를 들어 `token_refresh`와 같은 키를 기준으로 관리합니다.

2. 토큰 갱신 요청이 필요한 시점에 `LocalStorage`에 타임스탬프를 기록합니다.

3. 가장 먼저 기록에 성공한 탭이 대장(Leader)이 되어 실제 API 요청을 수행합니다.

4. 대장 탭이 응답을 받으면 `BroadcastChannel`을 통해 다른 탭들에게 “갱신 완료” 이벤트와 함께 결과 데이터를 직접 전송합니다.

5. 대장 탭이 네트워크 오류나 탭 닫힘으로 응답에 실패할 경우를 대비해, 일정 임대 시간(Lease Time)이 지나면 다시 선출이 가능하도록 설계했습니다.

## 5-4. 면접 대비 답변 스크립트

### Q. 여러 탭에서 동시에 토큰 갱신을 시도할 때 어떻게 해결했나요?

> 저는 **Single Flight Pattern**을 응용하여 **BroadcastChannel과 LocalStorage** 조합으로 경합 문제를 해결했습니다.
>
> 당시 프로젝트는 Safari를 포함한 광범위한 브라우저 호환성이 중요했고, 대시보드 성격상 탭을 수십 개씩 띄우는 환경이 아니었기에 SharedWorker보다는 리소스 효율이 좋은 이 방식을 선택했습니다.
>
> 구체적인 로직은 이렇습니다. 토큰 갱신이 필요한 시점에 각 탭은 LocalStorage를 확인합니다. 선착순으로 시점이 기재된 순간, 해당 시점으로부터 일정 시간 이내의 다른 요청들은 대기 상태로 들어갑니다.
>
> 선출된 대장 탭이 요청을 성공하면 그 결과를 BroadcastChannel로 공유하여 모든 탭이 최신 토큰을 동기화하게 했습니다.
>
> 추가적인 고도화를 고민한다면, 극히 드문 확률로 발생하는 완전 동시 진입, 즉 Race Condition을 방지하기 위해 각 탭의 요청에 **랜덤 지연(Jitter)**을 추가하여 경합 확률을 수학적으로 더 낮추는 방향으로 개선할 수 있다고 생각합니다.

## 5-5. BroadcastChannel API 학습 포인트

`BroadcastChannel API`는 동일한 Origin, 즉 같은 도메인을 가진 모든 브라우징 컨텍스트 간에 양방향 통신을 가능하게 하는 API입니다.

탭, 창, iframe 간 통신에 사용할 수 있으며, `postMessage()`로 메시지를 보내고 `onmessage` 이벤트로 메시지를 수신합니다.

Broadcast 방식이기 때문에 메시지를 보낸 자신을 제외한 모든 탭에 전달됩니다.

또한 메모리 누수를 방지하기 위해 사용 후에는 반드시 `channel.close()`가 필요합니다.
